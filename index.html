Task visibility — who sees what
Two-pass filtering in useRealtimeTasks.ts, on top of RLS
Pass 1 — Space filtering
1
Fetch all workspace spaces (RLS: all members can SELECT)
2
Owner/CEO/Admin/Dept Head/Manager → see ALL spaces
3
Staff/Intern/Client → only spaces where hidden_for doesn't include them AND space belongs to their department (or has no department)
Pass 2 — Task filtering
1
hidden_for includes user → always hidden for everyone
2
Owner/CEO/Admin/Dept Head/Manager → see ALL tasks
3
Staff/Intern/Client → task visible if: in allowed space, OR matches user's dept, OR user is in task_assignees (assignment always overrides space restriction)
RLS UPDATE policy: can_manage_tasks() OR is_task_assignee() — staff and interns can update tasks assigned to them even if they couldn't create them.

Admin protection rules
Admins can't remove other adminsOnly owner can remove/change admin roles
Owner role is lockedCannot be changed via normal role UI — only Transfer Ownership bypasses this
Enforcement layerFrontend only — not backed by RLS in v1
Admin removal protection is frontend-only. A malicious admin could call the Supabase REST API directly since the RLS UPDATE policy checks is_workspace_admin() equally for all three top-level roles. Needs RLS enforcement in v2.
Known gaps, inconsistencies, and v2 items
CEO excluded from Settings Workspace + Automations tabs
Check is userRole === "admin" || userRole === "owner" — CEO not included. CEO has full permissions everywhere else but cannot access workspace rename, danger zone, or automation rules. Inconsistent with the rest of the system.
Admin removal protection is frontend-only
The "only owner can remove admins" rule lives only in WorkspaceMembersSection.tsx. The RLS UPDATE policy allows any is_workspace_admin() user to modify any member row. Should be moved to RLS in v2.
Finance billing RLS is app-layer only
billing_records SELECT is open to all workspace members. The finance privilege is a page-access gate only. Any workspace member with a valid session token can query billing data directly.
Legacy roles (finance / hr / qc) have zero permissions
rolePermissions assigns [] to these three roles. They display as "Staff" but a user assigned one can't even add a comment or view their own tokens. Should be aliased to "staff" permissions or removed from the enum.
Automation rules stored in browser only
Stored in local/browser storage, not the DB. Per-browser, not shared across the team, lost on storage clear. Not server-persisted in v1.
canChatWithClients / canChatWithAdmins — defined but not enforced
Both flags computed correctly in AuthContext. Messages page does not gate DM creation on them — any role can DM any role in v1. Scaffolded for v2.
What works well
• Onboarding guard is truly fail-closed (blocks during loading — no content flash)
• Assigned tasks are ALWAYS visible even if the space is restricted
• 3-tier privilege resolution (role → department → explicit grant) is clean and extensible
• Role assignment hierarchy prevents privilege escalation
• DB helper functions are SECURITY DEFINER + STABLE — safe against RLS recursion and efficient
• Workspace creator correctly gets 'owner' role (fixed via migration in v1)
